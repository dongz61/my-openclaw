# Workload Notes

这份文档用于持续整理 OpenClaw 在 SWE-bench 场景下的单 agent workload。

当前目标不是追求自动化分类，而是先基于已有样本做人工归纳，逐步形成稳定的阶段划分、典型行为模式和优化信号。

后续每分析完新的样本，都继续修改这份文件。

## 当前观察来源

- `experiment/swebench/outputs/runs/*/run_report.md`
- `experiment/swebench/outputs/runs/*/summary.json`
- `experiment/swebench/outputs/reports/**/*.json`
- prompt 与 evaluator 输出

## 当前单 agent 核心阶段

### 1. 问题定位

目标：

- 根据 issue 描述快速锁定候选文件和候选函数

典型行为：

- 用 `exec` 搜文件、搜符号、搜测试入口
- 少量 `read` 打开最相关源码

当前表现：

- 这一步往往很快
- 成功样本通常能较早进入正确文件
- 如果一开始定位偏了，后续成本会明显膨胀

### 2. 问题认知建立

目标：

- 结合 issue 描述、代码阅读和运行结果，形成对“真正问题”的认知

典型行为：

- 读取局部实现
- 运行最小 Python 片段
- 对某个 API、控制分支或数据结构行为做验证
- 逐步形成一个模型自己相信的问题定义

当前表现：

- 这一阶段推理价值高，但实际步数不一定多
- 如果模型能在这里建立正确的问题认知和因果链，后续修复通常比较顺
- 如果这一阶段的问题认知出现偏斜、泛化或过早收敛，后续整条轨迹都可能被带偏

### 3. 主修复实现

目标：

- 对目标源码做最小修改，修复核心 bug

典型行为：

- 回读更大范围上下文
- `edit` 修改实现
- 再次 `read` 确认修改位置

当前表现：

- 在成功样本里，这一步有时反而很短
- 真正的核心修复未必复杂，复杂的是前后的验证链路

### 4. 回归测试设计

目标：

- 为当前 bug 增加或调整测试
- 确保 patch 有明确的验证目标

典型行为：

- 搜现有测试结构
- 阅读相近测试
- `edit` 添加测试
- 跑目标测试

当前表现：

- 如果测试加得准确，resolved 概率会显著提升
- 如果测试只覆盖“相邻问题”而不是 benchmark 的 FAIL_TO_PASS，用力很多也可能 unresolved

### 5. 运行相关测试

目标：

- 运行与当前修复直接相关的测试或脚本，确认修复成立且无明显回归

典型行为：

- `exec` 跑 Python 片段、目标测试、相关测试文件和更广范围相关测试
- 根据运行结果继续调整实现或测试

当前表现：

- 这是目前最重、最容易拉长轨迹的一段
- 很多 `exec` 成本来自这里，而不是来自真正的 bug 修复

### 6. 收尾与清理

目标：

- 确认最终 patch 干净、范围可控

典型行为：

- `git diff`
- `git status`
- 回退无关修改
- 再次检查最终工作树

当前表现：

- 成功样本通常会在这里收缩无关改动
- 如果模型没能清理验证阶段引入的噪声，最终 patch 容易被污染

## 横跨阶段的辅助行为

### 搜索测试与补齐上下文

目标：

- 搜源码、搜测试、搜数据文件、搜函数定义
- 判断测试应该落在哪
- 补齐当前阶段需要的局部上下文

典型行为：

- 大量 `exec` 搜测试、搜数据文件、搜相邻逻辑
- 穿插 `read` 阅读已有测试和实现代码

当前表现：

- 它不是一个边界清晰的独立阶段
- 更像横跨定位、测试设计和验证阶段的辅助行为
- 这是 `exec` 调用膨胀的重要来源之一
- 这部分成本往往不来自“修复逻辑本身”，而来自“决定如何验证、在哪里验证”

### 环境摩擦

目标：

- 记录运行、构建、依赖或版本问题对主轨迹造成的干扰

典型行为：

- `pip install`
- `setup.py` / build 相关命令
- 依赖版本调整
- 为了让运行继续而做的兼容修补

当前表现：

- 它不是只出现在某一个固定阶段
- 在问题认知建立和运行相关测试阶段尤其常见
- 很容易拉长轨迹，并把注意力从主问题上带开

## 当前初步行为模式

### A. 短路径成功

特征：

- 定位快
- 问题认知建立快
- edit 次数少
- 验证链路短

信号：

- 工具调用总数低
- `exec` 虽然仍常见，但不会长时间循环

### B. 验证主导型

特征：

- 主修复其实不复杂
- 大量成本花在测试定位、脚本验证、回归确认

信号：

- `exec` 数量高
- timeline 中后段充满测试命令和临时验证脚本

### C. 环境噪声型

特征：

- 运行验证时撞到依赖、构建、Python 版本兼容问题
- 模型开始修改与主问题无关的文件

信号：

- 出现额外的兼容修补
- 出现 build/setup/install 命令
- 后续需要回退无关修改

### D. 测试错位型

特征：

- 模型补了测试，也做了不少验证
- 但最终没有命中 benchmark 真正关心的 FAIL_TO_PASS

信号：

- agent 自己总结看起来合理
- evaluator 仍显示目标测试失败

## 6938 当前结论

样本：`astropy__astropy-6938`

当前判断：

- 属于“成功样本”
- 主问题定位准确
- 问题认知建立准确
- 主修复路径较短
- 中后段出现明显环境噪声
- 最终通过回退无关修改保持了 patch 干净

对应模式：

- 以“验证主导型”为主
- 同时带有“环境噪声型”特征

一句话总结：

- 这不是“修 bug 很难”，而是“把一个并不复杂的修复验证并干净落地，代价很高”

## 暂不做的事

当前先不急于：

- 自动阶段分类
- 自动 workload 压缩
- 仅靠命令字符串做强规则打标

原因：

- 现阶段样本量还不大
- 错误分类比人工慢一点更危险
- 先把人工 taxonomy 稳定下来更重要
